# 渠道加权随机算法解读

## 概述

New-API 实现了**两级渠道选择机制**：优先级（Priority）→ 权重随机（Weight）。当存在多个相同优先级的渠道时，根据配置的权重进行加权随机选择，确保高权重渠道获得更多流量。

## 核心文件

| 文件 | 职责 |
|------|------|
| `model/channel_cache.go` | 内存缓存模式下的加权随机选择（带平滑因子） |
| `model/ability.go` | 数据库直查模式下的加权随机选择（基础算法） |
| `middleware/distributor.go` | 请求分发入口，决定何时选择渠道 |

## 算法一：内存缓存模式（`channel_cache.go`）

### 函数签名
```go
func GetRandomSatisfiedChannel(group string, model string, retry int) (*Channel, error)
```

### 执行流程

```
1. 按 group + model 从内存索引 group2model2channels 中找到候选渠道 ID 列表
2. 按 Priority 分组，retry 参数决定使用第几优先级的渠道
3. 筛选出该优先级的所有渠道 → targetChannels
4. 计算总权重 sumWeight
5. 应用平滑因子（smoothingFactor / smoothingAdjustment）
6. rand.Intn(totalWeight) 生成随机数
7. 遍历 targetChannels，逐个减去权重，命中者胜出
```

### 平滑因子机制

这是该算法的核心亮点，解决了**小权重区分度不足**的问题：

```go
smoothingFactor := 1
smoothingAdjustment := 0

if sumWeight == 0 {
    // 场景：所有渠道权重都为 0（用户未配置权重）
    // 策略：每个渠道赋予等效权重 100，实现均匀随机
    sumWeight = len(targetChannels) * 100
    smoothingAdjustment = 100
} else if sumWeight/len(targetChannels) < 10 {
    // 场景：平均权重 < 10（权重差异太小）
    // 策略：放大 100 倍，拉开差距
    smoothingFactor = 100
}

totalWeight := sumWeight * smoothingFactor
randomWeight := rand.Intn(totalWeight)

for _, channel := range targetChannels {
    randomWeight -= channel.GetWeight()*smoothingFactor + smoothingAdjustment
    if randomWeight < 0 {
        return channel, nil
    }
}
```

### 示例

**场景 1：所有权重为 0（3 个渠道）**
```
smoothingAdjustment = 100, smoothingFactor = 1
totalWeight = 3 * 100 = 300
randomWeight ∈ [0, 300)

每个渠道有效权重 = 0*1 + 100 = 100
→ 每个渠道命中概率 = 100/300 = 33.3%
```

**场景 2：权重差异小（渠道 A=3, B=5, C=2）**
```
平均权重 = (3+5+2)/3 = 3.33 < 10 → smoothingFactor = 100
totalWeight = 10 * 100 = 1000
randomWeight ∈ [0, 1000)

A: 3*100 = 300 → 30%
B: 5*100 = 500 → 50%
C: 2*100 = 200 → 20%
```

**场景 3：权重差异大（渠道 A=100, B=200）**
```
平均权重 = 150 >= 10 → smoothingFactor = 1（不放大）
totalWeight = 300
randomWeight ∈ [0, 300)

A: 100 → 33.3%
B: 200 → 66.7%
```

## 算法二：数据库直查模式（`ability.go`）

### 函数签名
```go
func GetChannel(group string, model string, retry int) (*Channel, error)
```

### 执行流程

```
1. 构建查询：按 group + model + enabled + priority 过滤 ability 表
2. 按 weight DESC 排序获取符合条件的 ability 列表
3. 加权随机选择：
   - 每个 ability 的基础权重 = Weight + 10（保底权重 10）
   - 累加所有权重 → weightSum
   - rand.Intn(weightSum) 生成随机数
   - 遍历减法匹配
4. 根据选中的 ability.ChannelId 查询完整 Channel 对象
```

### 核心代码

```go
weightSum := uint(0)
for _, ability_ := range abilities {
    weightSum += ability_.Weight + 10  // 每个渠道保底 +10
}

weight := common.GetRandomInt(int(weightSum))
for _, ability_ := range abilities {
    weight -= int(ability_.Weight) + 10
    if weight <= 0 {
        channel.Id = ability_.ChannelId
        break
    }
}
```

### 与算法一的差异

| 对比项 | 内存缓存模式 | 数据库直查模式 |
|--------|-------------|---------------|
| 保底权重 | 0 时整体赋予 100 | 每个渠道固定 +10 |
| 平滑因子 | 有（平均<10 时放大 100 倍） | 无 |
| 性能 | 内存读取，极快 | 每次查 DB |
| 适用场景 | `common.MemoryCacheEnabled = true` | 默认模式 |

## 调用入口：`middleware/distributor.go`

请求分发时的渠道选择流程：

```
用户请求 → Distribute() 中间件
  │
  ├─ 是否指定了 channel_id？
  │   └─ 是 → 直接使用指定渠道
  │
  ├─ 是否有 affinity（亲和性）记录？
  │   └─ 是 → 优先使用上次成功的渠道
  │
  └─ 否 → CacheGetRandomSatisfiedChannel()
          │
          ├─ 内存缓存开启？
          │   ├─ 是 → channel_cache.go 的加权随机
          │   └─ 否 → ability.go 的加权随机
          │
          └─ 返回选中渠道 → SetupContextForSelectedChannel()
```

### 关键代码

```go
// middleware/distributor.go:137
channel, selectGroup, err = service.CacheGetRandomSatisfiedChannel(&service.RetryParam{
    Ctx:        c,
    ModelName:  modelRequest.Model,
    TokenGroup: usingGroup,
    Retry:      common.GetPointer(0),
})
```

## 权重 vs 优先级 的关系

```
Priority（优先级）: 决定「谁能被选中」
  - 高优先级渠道优先尝试
  - retry 参数递减时切换到低优先级
  - 类似于「主备」关系

Weight（权重）: 决定「同优先级下谁被选中的概率大」
  - 仅在相同优先级的多个渠道间生效
  - 权重越高，命中概率越大
  - 类似于「负载均衡」关系
```

### 示例

```
渠道 A: Priority=10, Weight=80
渠道 B: Priority=10, Weight=20
渠道 C: Priority=5,  Weight=100

首次请求（retry=0）:
  → 选择 Priority=10 的渠道 → A 或 B 加权随机
  → A 概率 80%, B 概率 20%

A/B 都失败后（retry=1）:
  → 降级到 Priority=5 → 选择 C
```

## 权重为 0 的特殊处理

当渠道权重为 0 时，**不会**被排除，而是：

- **内存缓存模式**：所有 0 权重渠道等效为权重 100，均匀分配
- **数据库直查模式**：每个渠道保底 +10，即权重 0 实际为 10

这确保了即使管理员未配置权重，系统也能正常工作。

---

## 注意事项

1. **权重是相对值**：权重 100 和 200 的效果等同于 1 和 2
2. **权重 0 不等于禁用**：禁用需设置 `Status != Enabled`
3. **平滑因子可能改变预期比例**：当平均权重 < 10 时，实际比例会被放大
4. **内存缓存延迟**：渠道权重修改后，需等待 `SyncChannelCache` 周期同步（默认频率可配置）
